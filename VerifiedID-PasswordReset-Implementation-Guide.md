# Microsoft Entra Verified ID for Student Password Reset

## Purpose
This guide explains how to:
1. Run the .NET Verified ID helpdesk sample locally.
2. Adapt it for student identity verification before password reset.
3. Plan a safe path to campus-wide production deployment.

## Architecture decision
Use the .NET helpdesk pattern as the baseline, then productionize it:
- Request presentation of `VerifiedEmployee` with Face Check.
- Validate callback result and confidence score.
- Map verified user to your student identity record.
- Trigger controlled password-reset workflow (not direct reset from browser action).

## Local setup (hands-on)

### 1. Add user-local .NET 8 to your shell
Add this to `~/.zshrc`:

```bash
export DOTNET_ROOT="$HOME/.dotnet"
export PATH="$HOME/.dotnet:$PATH"
```

Then reload shell:

```bash
source ~/.zshrc
dotnet --list-sdks
```

Expected: `8.0.x` appears.

### 2. Open the right sample

```bash
cd /Users/maramirez/repos/active-directory-verifiable-credentials-dotnet/6-woodgrove-helpdesk
```

### 3. Configure app settings for local dev
Create `appsettings.Development.json` (do not commit secrets). Start from `appsettings.json` and set:
- `VerifiedID:TenantId`
- `VerifiedID:DidAuthority`
- `VerifiedID:ManagedIdentity=false` (for local)
- `VerifiedID:ClientId`
- `VerifiedID:ClientSecret`

Notes:
- In Azure App Service production, switch back to `VerifiedID:ManagedIdentity=true`.
- Keep `VerifiedID:scope` as `3db474b9-6a0c-4840-96ac-1fceb342124f/.default`.

### 4. Register local app permissions
For local (client secret) testing, register an app in Entra and grant Verified ID Request Service application permissions.
- Baseline permission commonly used: `VerifiableCredential.Create.All`
- If you split by least privilege, use only presentation permissions for verifier-only app.

### 5. Ensure callback endpoint is publicly reachable
Verified ID callbacks must hit a public HTTPS URL.
For local testing, use ngrok (or deploy temporarily to App Service if ngrok is blocked).

Example:

```bash
ngrok http https://localhost:5001
```

Important:
- Open the app via the **ngrok HTTPS URL** in your browser, not `localhost`, so callback URLs generated by the app are public.

### 6. Run the sample

```bash
dotnet restore
DOTNET_ENVIRONMENT=Development dotnet run
```

Expected:
- App starts on `https://localhost:5001`
- Home page loads
- QR/presentation flow can be initiated

### 7. End-to-end functional test
1. Issue `VerifiedEmployee` via `https://myaccount.microsoft.com` for a test user.
2. In the sample, start verification and scan QR code with Microsoft Authenticator.
3. Complete Face Check.
4. Confirm callback reaches app and status transitions to `presentation_verified`.
5. Confirm returned claims include identity attributes needed for your helpdesk lookup (email/displayName by default).

## Production target design (campus)

### Reference flow
1. Student starts helpdesk verification (web, chat, or assisted flow).
2. System creates presentation request with Face Check and expected issuer DID.
3. Verified ID callback returns `presentation_verified`.
4. Backend validates:
   - credential type (`VerifiedEmployee` or your custom type)
   - issuer DID
   - linked domain validation
   - confidence threshold policy
5. Backend writes immutable audit event and opens/updates helpdesk case.
6. Backend triggers password-reset action via approved path:
   - Prefer issuing Temporary Access Pass (TAP) / controlled reset flow.
   - For on-prem AD, call a secured internal reset service (not from front-end).

### Production hardening required (before rollout)
The sample is demo-quality and needs upgrades:
- Replace in-memory state cache with Redis/DB (sample uses memory cache).
- Replace per-startup random callback API key with stable, rotated secret (Key Vault).
- Add operator authentication/authorization (RBAC) for helpdesk staff.
- Add anti-automation controls: rate limiting, abuse monitoring, replay protection.
- Implement structured audit logs with PII minimization and retention policy.
- Add resilience: retries, dead-letter handling, idempotency for callback processing.
- Add monitoring: App Insights dashboards, alerting, SLOs.

## Campus rollout plan

### Phase 0: Design and governance (2-4 weeks)
- Define policy: who is eligible, what confidence threshold is acceptable, fallback process.
- Decide account scope: Entra-only, hybrid, or AD-only populations.
- Define legal/compliance requirements (privacy notice, log retention, operator access).
- Complete threat model and runbook approvals.

### Phase 1: Pilot (4-6 weeks)
- Limited group (e.g., IT staff + small student cohort).
- Use App Service + Managed Identity in non-prod tenant/subscription.
- Measure: verification success rate, false rejection rate, average handle time, reset completion time.
- Finalize support scripts and training.

### Phase 2: Controlled expansion (4-8 weeks)
- Expand to selected colleges/departments.
- Integrate with ticketing/service desk workflow.
- Add dashboards and weekly review cadence.
- Tune threshold and fallback decisions based on observed outcomes.

### Phase 3: Campus-wide go-live
- Production change window and communications.
- Progressive traffic ramp-up.
- Post-launch hypercare period with daily health checks.

## Important product considerations to decide early
1. Credential type:
   - `VerifiedEmployee` can work quickly for workforce-style users.
   - You may need a custom student credential model if claim semantics differ.
2. User population:
   - If students are in an EDU tenant, check setup path carefully.
3. Recovery path:
   - If your true goal is total-loss account recovery, evaluate Entra Account Recovery (Preview) in parallel.
